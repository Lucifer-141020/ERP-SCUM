## 3. 常用验证命令 / 一键验证总脚本

每次修改完功能、整理完代码、准备交付复验前，**优先跑这一条命令代替手工逐个验证**：

```bash
node scripts/verify-all.js
```

### 这是什么

这是**本地一键检查命令**，不是部署命令。它的作用是：

- 自动检查生产页和 frontend/ 拆分源码是否同步（`check-frontend-sync.js`）
- 自动把 frontend/ 合并成预览文件，检查合并过程是否正确（`build-frontend-preview.js`）
- 自动跑 HTML 和 main.js 的语法检查
- 自动跑后端测试
- 自动跑全部 16 个专项测试

### 什么时间运行

- ✅ 每次修改功能后
- ✅ 每次整理代码后
- ✅ 每次交付复验前

### 通过标准

看到下面这行才算通过：

```
全部验证通过
```

如果中间任意一步失败，脚本会马上停下来，不会假装成功。

### 它不是

- ❌ 不是部署命令，**不会上传服务器**
- ❌ **不会修改数据库**
- ❌ **不会正式发布网站**
- ❌ **不会覆盖生产页面**

### 特殊情况

如果你只想检查同步状态（最快），可以只跑：

```bash
node scripts/check-frontend-sync.js
```

但完整交付前仍建议跑 `node scripts/verify-all.js`。

## 9. Git 当前规则

“Git 暂时不动”不是说永远不用 Git。

它的意思是：

```text
不要擅自 git commit
不要擅自 git push
不要擅自 git pull
不要让服务器通过 Git 拉取当前本地半成品
```

原因：

- 当前 C 盘是验收副工作区，O 盘才是主版本和部署来源。
- GitHub 提交前仍需精确检查暂存区，不能使用 `git add -A`。
- 数据库、上传图片、环境文件和备份不能进入 GitHub。

只有用户明确说类似下面的话，才考虑 Git：

```text
这个版本定稿，可以提交 Git
```

即使用户这样说，也必须先：

1. 查看 `git status -sb`。
2. 说明准备提交哪些文件。
3. 运行必要检查。
4. 等用户再次确认。

不要乱用：

```bash
git reset --hard
git checkout -- .
git clean -fd
```

这些命令可能会删除本地修改或未提交文件。

## 15. 正式密码和管理员权限

正式密码在服务器部署时设置，不在本地代码里设置。

启动后端时通过环境变量设置：

```bash
NODE_ENV=production ADMIN_PASSWORD="YOUR_REAL_ADMIN_PASSWORD" JWT_SECRET="YOUR_RANDOM_LONG_SECRET" PORT=3000 node server.js
```

说明：

```text
ADMIN_PASSWORD：后台登录密码
JWT_SECRET：JWT 签名密钥，仅保存在服务器环境变量中
PORT：后端运行端口
```

后台密码建议：

```text
不少于 12 位
包含大小写字母、数字、符号
不要用生日、QQ、手机号、服务器名
不要发到群里
不要写进 GitHub
```

当前第一版逻辑是：

```text
知道后台密码的人就是管理员
```

第一阶段上线够用。后期如果需要，再升级多个管理员账号、不同权限、操作日志和修改记录。

### 18.4 配置 Nginx

Nginx 的作用是：让外网访问域名时，正确打开网页，并把接口请求转给后端。

参考配置：

```nginx
server {
  listen 80;
  server_name 你的域名.com www.你的域名.com;

  root /www/wwwroot/ERP-SCUM;
  index erp14-server-showcase.html;

  location / {
    try_files $uri $uri/ /erp14-server-showcase.html;
  }

  location /api/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  location /uploads/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

### 18.5 配置 HTTPS

上线后建议开启 HTTPS。

HTTPS 的作用是：让网址显示安全锁，保护登录密码和后台操作数据。

如果使用宝塔或 1Panel，可以在面板里申请免费证书。

申请成功后，访问地址应该是：

```text
https://你的域名.com
```

## 23. 常见问题

### 23.1 网站能打开，但后台保存失败

重点检查：

```text
后端是否运行
Nginx 是否正确转发 /api/
后台密码是否正确
服务器 data 目录是否有写入权限
```

### 23.2 图片上传成功但前台不显示

重点检查：

```text
backend/uploads 是否存在
Nginx 是否正确转发 /uploads/
图片文件是否真实存在
浏览器地址是否能直接打开图片
```

### 23.3 GitHub 上没有 data 和 uploads

这是正常的。

原因：

```text
backend/data 是运行数据
backend/uploads 是上传图片
```

它们不适合放 GitHub，应该放服务器并单独备份。

### 23.4 域名访问接口不对

当前网页会自动用当前域名作为后端地址。

所以正式域名下必须保证：

```text
https://你的域名.com/api/site-config
```

能访问到后端。

如果不能，通常是 Nginx 的 `/api/` 转发没配好。

## 27. 最重要的安全原则

- 不要把半成品当成上线版本。
- 不要把 Git 当前仓库当成最新线上状态。
- 不要把 `发布区\` 当成开发入口。
- 不要覆盖服务器运行数据和上传图片。
- 不要删除当前服务器稳定版前端备份。
- 不要在用户没确认前上传、重启、提交或推送。
