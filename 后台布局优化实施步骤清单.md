# 后台布局优化实施步骤清单

> 本清单用于 ERP14 SCUM 服务器网页后台长期优化。新的 Codex 对话必须先阅读本文件，再继续后台布局相关工作。

## 1. 当前确认的后台设计方向

### 1.1 使用人群

- 后台主要给二胖总裁和少数可信管理员使用。
- 暂不做复杂权限分级。
- 优先保证：看得懂、找得到、少误操作、方便长期维护。

### 1.2 主菜单结构

后台主菜单采用运营型结构：

1. 总览
2. 网站基础信息
3. 首页展示
4. 活动管理
5. 玩家建议
6. 图片库
7. 备份恢复
8. 操作日志

### 1.3 各菜单确认结论

| 菜单 | 方向 |
| --- | --- |
| 总览 | 待办优先，轻量总览，每块只显示数量和最近 3-5 条 |
| 网站基础信息 | 表单编辑，管理服务器名、IP、QQ群、公告、赛季文案、基础标签 |
| 首页展示 | 模块列表编辑；活动不在这里直接编辑，只控制展示规则 |
| 活动管理 | 保持当前流程和模式；活动类型只保留长期活动、限时活动 |
| 玩家建议 | 保持目前流程和模式，只做小优化 |
| 图片库 | 只做小优化，保留批量管理能力 |
| 备份恢复 | 安全优先，导入/重置前自动备份，重置必须强确认 |
| 操作日志 | 轻量日志，显示最近操作、筛选、高风险标记、操作详情 |

## 2. 不允许擅自扩大范围

本轮后台布局优化不做以下事情：

- 不重做整站前台视觉。
- 不重构数据库结构。
- 不大拆 `erp14-server-showcase.html`。
- 不新增复杂权限系统。
- 不把活动改成公告活动、自定义类型或更多分类。
- 不把玩家建议改成看板或工单流。
- 不直接部署服务器，部署必须单独确认。

## 3. 第一阶段：后台骨架整理

### 目标

让后台入口更清楚，但不改变业务数据和原有保存逻辑。

### 任务

1. 备份当前 `erp14-server-showcase.html`。
2. 整理后台主菜单顺序为：
   - 总览
   - 网站基础信息
   - 首页展示
   - 活动管理
   - 玩家建议
   - 图片库
   - 备份恢复
   - 操作日志
3. 增加或整理“总览”页。
4. 总览页显示：
   - 待处理玩家建议数量
   - 正在报名的活动
   - 最近报名
   - 图片库状态
   - 最近备份状态
   - 最近操作日志

### 成功标准

- 登录后台后菜单顺序清楚。
- 原有功能入口没有丢失。
- 总览只做只读展示，不影响任何保存功能。
- 前台页面显示不受影响。

## 4. 第二阶段：首页展示和基础信息整理

### 目标

降低后台编辑难度，避免管理员手写复杂格式。

### 任务

1. 网站基础信息按表单分组。
2. 首页展示按模块分组：
   - 轮播图
   - 首页主文案
   - 首页特色
   - 首页数据卡
   - 玩家群入口
   - 活动展示规则
3. 活动内容仍统一在“活动管理”编辑。
4. 首页只控制活动展示规则，例如：
   - 是否显示活动入口
   - 显示几个活动
   - 是否优先显示报名中活动
   - 是否显示活动封面

### 成功标准

- 后台结构更清楚。
- 活动没有出现两个编辑入口。
- 保存后前台内容正常显示。

## 5. 第一优先小步：首页数据卡和首页特色卡表单化

### 背景问题

当前后台存在类似以下格式：

```text
users||服务器容量|128 人|中大型SCUM游戏社区
package|amber|经济与交易|玩家可以围绕物资、载具、服务和活动奖励形成交易关系，服务器更像一个小社区。|交易区,活动奖励
```

这种格式对程序好解析，但对管理员不友好，容易少写或多写 `|` 导致显示错误。

### 目标

只改后台编辑方式，不改前台展示、不改后端接口、不改底层数据格式。

### 任务

1. 定位“首页数据卡”的 textarea 和保存逻辑。
2. 定位“首页特色卡”的 textarea 和保存逻辑。
3. 新增解析函数，把旧格式拆成表单字段。
4. 新增拼接函数，保存时自动拼回旧格式。
5. 首页数据卡改成表单列表：
   - 图标
   - 颜色
   - 小标题
   - 大数字
   - 说明
   - 上移
   - 下移
   - 删除
   - 新增数据卡
6. 首页特色卡改成表单列表：
   - 图标
   - 颜色
   - 标题
   - 说明
   - 标签 1
   - 标签 2
   - 上移
   - 下移
   - 删除
   - 新增特色卡
7. 颜色只允许：
   - 默认
   - blue
   - amber
   - red
8. 图标为空时允许保存，但建议自动给默认值。
9. 标题、大数字等关键字段为空时给提示。

### 成功标准

- 管理员不需要手写 `|`。
- 旧数据打开后能正常显示为表单。
- 保存后仍兼容当前前台展示。
- 新增、删除、上移、下移可用。
- 刷新后台后数据仍在。

## 6. 第三阶段：活动、建议、图片库小优化

### 活动管理

- 保持现有活动分类和状态。
- 类型只保留：
  - 长期活动
  - 限时活动
- 状态保持：
  - 报名中
  - 进行中
  - 已结束
  - 已颁奖
  - 常驻
- 不新增公告活动。
- 不做自定义类型。

### 玩家建议

- 保持目前流程和模式。
- 只优化：
  - 文案
  - 按钮
  - 状态颜色
  - 删除/拒绝前确认

### 图片库

- 保留当前批量管理。
- 优化上传区、筛选区、图片信息、复制地址、批量删除确认。

## 7. 第四阶段：高风险功能加固

### 备份恢复

- 导入前自动备份当前数据。
- 重置前自动备份当前数据。
- 导入时明确显示会覆盖哪些模块。
- 重置必须二次确认。
- 重置建议要求输入 `确认重置`。
- 页面明确提示：普通代码更新不要覆盖服务器运行数据。

### 操作日志

- 轻量优化。
- 显示最近操作、筛选、高风险标记、操作详情。
- 删除、导入、重置、批量删除要醒目标记。
- 暂不做字段级详细审计。

## 8. 每次实施前必须做的事

1. 阅读 `AGENTS.md`。
2. 阅读本文件。
3. 阅读 `更新日志.md`。
4. 备份即将修改的文件。
5. 明确本次只做哪一小步。
6. 不顺手重构无关代码。

## 9. 每次实施后必须验证

至少执行：

1. HTML 内嵌 JS 语法检查。
2. 后端测试。
3. 浏览器打开前台确认页面不白屏。
4. 浏览器打开后台确认目标功能可用。
5. 保存后刷新，确认数据仍在。
6. 确认前台展示不坏。

## 10. 每次实施后必须记录

每次修改、更新、优化、删除、备份、发布准备、部署操作，都必须写入 `更新日志.md`。

记录内容至少包括：

- 日期时间
- 操作人
- 操作类型
- 涉及文件
- 改动内容
- 备份位置
- 验证结果
- 是否部署
- 后续注意事项
