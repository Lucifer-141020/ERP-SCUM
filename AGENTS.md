本文件是 `SCUM网页` 项目的唯一总则。Codex 在处理本项目任何任务时，必须优先遵守本文件。

以后本项目的所有记忆迭代、规则补充、经验复盘、长期要求，都只更新这个文件：

```text
O:\AI\Open AI_Codex\SCUM用户网页\AGENTS.md
```

## 0. 工作区分工与同步边界

本项目采用“C 盘副工作区、O 盘主工作区”的双工作区流程：

- C 盘副工作区：`C:\Users\Administrator\Desktop\SCUM用户网页`
  - 用于日常代码修改、测试、预览、差异分析和安全整改。
  - 主要用于与 Workbuddy 协作执行具体任务；Codex 负责总协调、任务拆解、范围控制、验收判断、证据汇总和最终交付把关。
  - 默认所有开发操作先在这里完成。
  - 可以使用临时数据库、临时端口和临时测试数据，但不得连接或覆盖正式数据。
- O 盘主工作区：`O:\AI\Open AI_Codex\SCUM用户网页`
  - 用于保存已验收的主版本、正式归档、最终发布包和部署准备材料。
  - 未通过验收的 C 盘改动不得同步到这里。
  - O 盘内容不得被 C 盘测试结果自动覆盖。

### 同步规则

1. 先在 C 盘完成一个独立事项。
2. 在 C 盘完成对应测试、差异检查、备份和更新日志记录。
3. 明确列出准备同步的文件、排除的文件和回滚方式。
4. 只有用户明确确认同步后，才把已验收文件从 C 盘同步到 O 盘。
5. 同步后必须在 O 盘重新检查文件存在性、SHA256、语法和必要测试。
6. 不使用 `git add -A` 或整目录覆盖同步；只复制已验收的正式源码和文档。

### 部署规则

- 部署、上传服务器、生成正式发布包、重启 PM2 或修改 Nginx，只能以 O 盘主工作区为来源。
- C 盘不得直接部署，也不得作为服务器上传源。
- 当前阶段默认“不部署”；除非用户明确确认，不执行上传、重启、覆盖正式数据或生产页面操作。
- 正式数据库、上传图片、服务器环境配置和生产页面备份属于 O 盘主工作区的正式资产，不能从 C 盘测试数据代替。

### 路径判断优先级

- 用户说“开发、修改、测试、预览”：默认使用 C 盘副工作区。
- 用户说“同步、归档、正式发布、部署”：先在 C 盘完成验收，再使用 O 盘主工作区。
- 发现 C 盘与 O 盘内容不一致时，先报告差异来源，不自动选择一边覆盖另一边。

不要为本项目再新建第二个 `AGENTS.md`。如果发现其他位置存在同名规则文件，必须先提醒用户，再询问是否合并进本文件。

## 3. 操作前规则

对本项目进行任何改动前，必须先阅读：

1. `AGENTS.md`

## 4. 简洁优先

用最少的代码解决问题，不做无关扩展。

- 不做超出需求的功能。
- 不为一次性代码做抽象。
- 不做未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景增加复杂错误处理。
- 如果一个需求 50 行能解决，不要写成 200 行。

## 5. 精准改动

只修改与当前任务直接相关的内容。

- 不顺手“改进”相邻代码、注释或格式。
- 不重构没坏的东西。
- 遵循项目现有风格，即使个人不喜欢。
- 不删除无关文件。
- 不改变数据库结构，除非老大明确授权。
- 不改变服务器接口，除非当前任务必须这样做并已提前说明风险。
- 不直接部署服务器，部署必须单独确认。
- 如果发现无关死代码，只提醒用户，不主动删除。
- 如果自己的改动导致未使用的导入、变量或函数，要清理自己造成的问题。
- 每一行改动都应该能直接追溯到用户请求。

### 12.3 高风险操作

高风险操作包括：

- 删除文件或数据
- 批量替换

高风险操作必须做到：

- 先备份，必要时提醒老大做服务器快照。
- 得到老大明确确认后才能执行。
- 每一步完成后先验证，再继续下一步。

## 14. 错误止损规则

出现以下情况时，必须暂停继续改动，先分析原因：

- 同一个问题连续验证失败 2 次。
- 页面白屏。
- 后台无法登录或无法保存。
- 保存后数据丢失或显示异常。
- 后端接口返回异常。
- 测试从通过变成失败。
- 服务器访问失败。
- 备份、恢复、上传、部署结果不确定。

止损时必须做到：

- 先保留现场，不继续扩大修改范围。
- 说明已经做过哪些操作。
- 判断是否需要回滚到最近备份。
- 如果无法确定原因，必须直接说明不确定，不能继续猜。
- 给出下一步最小验证动作，而不是连续尝试多个方向。

## 15. 服务器面板类操作复盘与强制流程

本节来自一次宝塔面板端口修改失败后的复盘。问题起点只是“默认端口 8888 有风险”和“HTTP 访问面板不安全”，但处理过程中没有先做快照、没有及时止损，导致用户连续执行多轮面板端口、安全入口、限制取消、修复和更新操作，最后宝塔面板出现 `nginx 404` / `HTTP 502`，面板无法正常登录。

以后遇到宝塔、aaPanel、服务器防火墙、安全组、面板端口、面板 SSL、安全入口这类高风险操作，必须遵守以下规则：

- 动手前先明确当前可用入口，至少确认一次 `bt default` 或面板当前登录地址。
- 涉及面板端口、防火墙、安全入口、SSL、覆盖安装、重装、修复面板前，必须先提醒用户做云服务器快照。快照就是服务器当前状态的备份。
- 不允许在没有快照的情况下连续引导用户执行多轮修复、更新、重启、改入口、改端口等操作。
- 修改端口的正确顺序必须是：先放行新端口，再确认新端口规则存在，再修改面板端口，再用新地址验证，最后才关闭旧端口。
- 如果 `bt default` 显示端口没有变化，说明端口修改没有成功，不要假设新端口已经生效。
- 如果浏览器报 `502`，不要马上归因于 HTTP/HTTPS；必须先用服务器本机命令验证，例如 `curl -i http://127.0.0.1:端口/入口`。
- 如果本机 `curl` 连续返回 `nginx 404`，并且 `ss -lntp` 显示面板端口确实由 `BT-Panel` 监听，应尽快判断为宝塔面板内部路由或程序配置异常，不要继续反复改安全入口、取消限制或猜测浏览器问题。
- 如果 `16 修复面板` 和 `34 更新面板/更新到最新版` 都不能解决，应立即停止试错，给出两条路线：先做快照后找官方支持，或先做快照后执行官方覆盖安装面板。
- 覆盖安装或重装面板前，必须明确说明风险：通常不删除网站文件和数据库，但会改动面板程序本身，因此必须先做快照。
- 任何日志、截图、命令输出中出现 `access_key`、`secret_key`、`token`、`cookie`、`session`、账号密码等敏感信息时，必须立即提醒用户打码，并建议后续轮换密钥。

本次错误的经验：

- 不要把“看起来可能是协议问题、入口问题、防火墙问题”的猜测当成结论。
- 连续两三次验证都指向同一类异常时，要收敛判断并止损，不能继续让用户试命令。
- 对新手用户，服务器面板类操作必须优先保护可回滚能力，而不是优先追求立刻修好。
- 如果无法确定修复路径，应该诚实说明边界，建议快照后找宝塔或云厂商官方支持，而不是继续扩大操作范围。

## 16. 复盘沉淀规则

凡是出现错误、回滚、部署失败、测试失败、页面异常、数据异常，都必须在 `更新日志.md` 中额外记录复盘。

复盘至少包括：

- 问题表现：实际发生了什么。
- 影响范围：影响了哪些页面、文件、数据或服务器功能。
- 直接原因：目前能确认的原因。
- 处理方式：做了什么修复或回滚。
- 验证结果：如何确认已经恢复。
- 避免方法：下次如何减少同类问题。

如果原因暂时无法确认，必须写“原因未确认”，并记录下一步建议，不允许把猜测写成结论。

## 17. 长期维护原则

- 宁可小步多次，也不要一次改太大。
- 每次只解决一个清楚的问题。
- 先保证可恢复，再追求优化速度。
- 先保护服务器数据，再更新页面代码。
- 先保护后台可用，再优化后台体验。
- 新对话必须通过 `AGENTS.md`、`后台布局优化实施步骤清单.md`、`更新日志.md` 继承上下文，不允许从记忆里猜项目状态。
